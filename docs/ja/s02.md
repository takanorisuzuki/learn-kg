# ナレッジグラフとは何か


> "エンティティと関係をつなぐことで、見えなかったものが見えてくる。"

## 問題

「ある顧客が報告したバグに、どのエンジニアが現在アサインされていて、そのエンジニアが同時に抱える他のクリティカルチケットは何か」。この問いにRDBで答えようとすると、こうなります。

```sql
-- 3段階の関係をたどるSQLクエリ
SELECT DISTINCT
    e.name AS engineer_name,
    t2.title AS other_critical_tickets
FROM customers c
JOIN tickets t1 ON c.id = t1.customer_id
JOIN bug_assignments ba ON t1.bug_id = ba.bug_id
JOIN engineers e ON ba.engineer_id = e.id
JOIN ticket_assignments ta ON e.id = ta.engineer_id
JOIN tickets t2 ON ta.ticket_id = t2.id
WHERE c.name = '顧客X'
  AND t1.type = 'bug'
  AND t2.priority = 'critical'
  AND t2.id != t1.id;
```

6テーブルのJOINです。さらに「そのエンジニアのチームのマネージャーは誰か」を加えると、JOIN数はさらに増えます。パフォーマンスの問題だけでなく、スキーマ変更のたびに既存クエリの修正が必要になります。

## 解決策

ナレッジグラフ（KG）は、知識を「ノード（実体）」と「エッジ（関係）」で表現します。2012年にGoogleがKGを発表した際の言葉がこの本質を表しています。

> **"Things, not strings."**（文字列ではなく、実体を）

同じクエリをNeo4jのCypherで書くとこうなります。

```cypher
MATCH (c:Customer {name: '顧客X'})-[:REPORTED]->(b:Bug)
      <-[:ASSIGNED_TO]-(e:Engineer)
      -[:WORKS_ON]->(t:Ticket {priority: 'critical'})
WHERE t <> b
RETURN e.name AS engineer_name, t.title AS other_critical_tickets
```

JOINは不要です。エッジをたどるだけです。「マネージャーも知りたい」という追加要件は、1行追加するだけで対応できます。

## 仕組み

### トリプル構造：知識の最小単位

KGの基本表現は「トリプル（Triple）」です。**主語 - 述語 - 目的語**の3要素で、あらゆる知識を表現します。Neo4jのCypherでは、このトリプルがそのままノードとリレーションシップに対応します。

- 鈴木一郎 **が所属する** 株式会社ABC → `(e:Employee)-[:WORKS_FOR]->(c:Company)`
- 鈴木一郎 **が担当する** プロジェクトAlpha → `(e:Employee)-[:ASSIGNED_TO]->(p:Project)`
- プロジェクトAlpha **の承認者は** 山田部長 → `(p:Project)-[:APPROVED_BY]->(m:Manager)`

この3つのトリプルがあれば「鈴木さんのプロジェクトの承認者のメールアドレス」をグラフのパスをたどるだけで答えられます。

### Neo4jでのノード・エッジ定義

```cypher
// ノードの作成（ラベル付き）
CREATE (e:Employee {
    id: 'emp-001',
    name: '鈴木一郎',
    email: 'suzuki@example.com'
})

CREATE (p:Project {
    id: 'project-alpha',
    name: 'プロジェクトAlpha',
    status: 'active'
})

// リレーションシップの作成
MATCH (e:Employee {id: 'emp-001'}), (p:Project {id: 'project-alpha'})
CREATE (e)-[:ASSIGNED_TO {allocation_rate: 0.8}]->(p)
```

`Employee`・`Project` というラベルと `ASSIGNED_TO` というリレーションシップタイプが型の役割を果たします。

## このセッションで変わること

**始める前の状態：**
- LLMが正確な事実に弱いことはわかった。しかし、その事実をどう格納するかのモデルがない
- 「担当バグを抱えるエンジニアのチームマネージャーは誰か」という問いが複雑なSQL結合を必要とする
- ナレッジグラフという言葉は知っているが、データ構造として説明できない

**このセッション後の状態：**
- ドメインの知識をCypherのノードとエッジで表現できる
- トリプル（主語-述語-目的語）がNeo4jのノード・リレーションシップに直接対応することを理解している
- 関係が複雑なクエリで、グラフ探索がSQL多段結合に取って代わる理由がわかる

## 試してみる

Neo4jがなくても、Pythonの`networkx`ライブラリでグラフ操作をすぐに試せます。

```python
import networkx as nx

# グラフの作成（有向グラフ）
G = nx.DiGraph()

# ノードの追加（属性付き）
G.add_node("鈴木一郎", type="Employee", email="suzuki@example.com")
G.add_node("株式会社ABC", type="Company", industry="SaaS")
G.add_node("プロジェクトAlpha", type="Project", status="active")
G.add_node("山田部長", type="Manager", email="yamada@example.com")

# エッジ（関係）の追加
G.add_edge("鈴木一郎", "株式会社ABC", relation="WORKS_FOR", since="2022-04-01")
G.add_edge("鈴木一郎", "プロジェクトAlpha", relation="ASSIGNED_TO", rate=0.8)
G.add_edge("プロジェクトAlpha", "山田部長", relation="APPROVED_BY")
G.add_edge("山田部長", "株式会社ABC", relation="MANAGES")
```

```python
# 2ホップ先までの関係をすべて表示
def show_neighborhood(G, start_node, max_hops=2):
    """指定ノードから2段階以内の関係をすべて表示"""
    visited = {start_node}
    current_level = {start_node}

    for hop in range(1, max_hops + 1):
        next_level = set()
        for node in current_level:
            for neighbor in G.successors(node):
                if neighbor not in visited:
                    edge = G.get_edge_data(node, neighbor)
                    print(f"  {'  ' * hop}{node} --[{edge['relation']}]--> {neighbor}")
                    next_level.add(neighbor)
                    visited.add(neighbor)
        current_level = next_level

show_neighborhood(G, "鈴木一郎")
# 出力:
#   鈴木一郎 --[WORKS_FOR]--> 株式会社ABC
#   鈴木一郎 --[ASSIGNED_TO]--> プロジェクトAlpha
#     プロジェクトAlpha --[APPROVED_BY]--> 山田部長
```

**実行してみよう：**

1. `pip install networkx` でインストール
2. 上記コードを実行して `show_neighborhood(G, "鈴木一郎")` の出力を確認する
3. 自分のユースケースに合わせてノードとエッジを追加してみる（例：チームメンバー、プロジェクト、担当製品など）
4. 「AさんはBプロジェクトに関わっているか」をグラフ探索で確認する
