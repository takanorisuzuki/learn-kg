# What Is a Knowledge Graph?


> "Connect entities and relationships, and you start to see what was invisible before."

## Problem

Consider this question: "Which engineer is currently assigned to the bug a customer reported, and what other critical tickets is that engineer working on at the same time?" Answering this with a relational database looks like this:

```sql
-- A SQL query that traverses three levels of relationships
SELECT DISTINCT
    e.name AS engineer_name,
    t2.title AS other_critical_tickets
FROM customers c
JOIN tickets t1 ON c.id = t1.customer_id
JOIN bug_assignments ba ON t1.bug_id = ba.bug_id
JOIN engineers e ON ba.engineer_id = e.id
JOIN ticket_assignments ta ON e.id = ta.engineer_id
JOIN tickets t2 ON ta.ticket_id = t2.id
WHERE c.name = 'Customer X'
  AND t1.type = 'bug'
  AND t2.priority = 'critical'
  AND t2.id != t1.id;
```

Six JOINs. Add "who is the manager of that engineer's team?" and the JOIN count grows further. Beyond performance issues, every schema change requires rewriting existing queries.

## Solution

A knowledge graph (KG) represents knowledge as **nodes (entities)** and **edges (relationships)**. When Google introduced the Knowledge Graph in 2012, they captured the essence in a phrase:

> **"Things, not strings."**

The same query in Neo4j's Cypher looks like this:

```cypher
MATCH (c:Customer {name: 'Customer X'})-[:REPORTED]->(b:Bug)
      <-[:ASSIGNED_TO]-(e:Engineer)
      -[:WORKS_ON]->(t:Ticket {priority: 'critical'})
WHERE t <> b
RETURN e.name AS engineer_name, t.title AS other_critical_tickets
```

No JOINs — just traversing edges. Adding "also show the manager" requires a single extra line.

## How It Works

### Triple Structure: The Smallest Unit of Knowledge

The fundamental representation in a KG is the **triple**: **subject - predicate - object**. Every piece of knowledge can be expressed with these three elements. In Neo4j's Cypher, every triple maps directly to a node or relationship.

- Alice Chen **works for** TechCorp → `(e:Employee)-[:WORKS_FOR]->(c:Company)`
- Alice Chen **is assigned to** Project Alpha → `(e:Employee)-[:ASSIGNED_TO]->(p:Project)`
- Project Alpha **is approved by** Bob Smith → `(p:Project)-[:APPROVED_BY]->(m:Manager)`

With these three triples, you can answer "What is the email address of the approver for Alice's project?" simply by following graph paths.

### Defining Nodes and Edges in Neo4j

```cypher
// Create nodes with labels
CREATE (e:Employee {
    id: 'emp-001',
    name: 'Alice Chen',
    email: 'alice@example.com'
})

CREATE (p:Project {
    id: 'project-alpha',
    name: 'Project Alpha',
    status: 'active'
})

// Create a relationship
MATCH (e:Employee {id: 'emp-001'}), (p:Project {id: 'project-alpha'})
CREATE (e)-[:ASSIGNED_TO {allocation_rate: 0.8}]->(p)
```

The labels `Employee` and `Project` along with the relationship type `ASSIGNED_TO` serve as the type system.

## Try It

You can start exploring graph operations right away using Python's `networkx` library — no Neo4j required.

```python
import networkx as nx

# Create a directed graph
G = nx.DiGraph()

# Add nodes with attributes
G.add_node("Alice Chen", type="Employee", email="alice@example.com")
G.add_node("TechCorp", type="Company", industry="SaaS")
G.add_node("Project Alpha", type="Project", status="active")
G.add_node("Bob Smith", type="Manager", email="bob@example.com")

# Add edges (relationships)
G.add_edge("Alice Chen", "TechCorp", relation="WORKS_FOR", since="2022-04-01")
G.add_edge("Alice Chen", "Project Alpha", relation="ASSIGNED_TO", rate=0.8)
G.add_edge("Project Alpha", "Bob Smith", relation="APPROVED_BY")
G.add_edge("Bob Smith", "TechCorp", relation="MANAGES")
```

```python
# Display all relationships up to 2 hops away
def show_neighborhood(G, start_node, max_hops=2):
    """Show all relationships within 2 hops from the given node"""
    visited = {start_node}
    current_level = {start_node}

    for hop in range(1, max_hops + 1):
        next_level = set()
        for node in current_level:
            for neighbor in G.successors(node):
                if neighbor not in visited:
                    edge = G.get_edge_data(node, neighbor)
                    print(f"  {'  ' * hop}{node} --[{edge['relation']}]--> {neighbor}")
                    next_level.add(neighbor)
                    visited.add(neighbor)
        current_level = next_level

show_neighborhood(G, "Alice Chen")
# Output:
#   Alice Chen --[WORKS_FOR]--> TechCorp
#   Alice Chen --[ASSIGNED_TO]--> Project Alpha
#     Project Alpha --[APPROVED_BY]--> Bob Smith
```
